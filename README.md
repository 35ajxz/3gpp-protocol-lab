# 3GPP Protocol Lab

面向 `TS 38.331 Rel-17` 的 3GPP 协议安全研究工程。

这个项目当前只做两件事，而且都要求结果可复验、可复现：

- 把真实规范整理成可信的协议模型。
- 基于协议状态机做逻辑漏洞挖掘与有状态模糊测试。

## 1. 项目定位

### 1.1 这个项目解决什么问题

RRC 协议安全研究里最难的部分，通常不是“能不能读懂规范”，而是“能不能把规范转成工程上真正能用的测试资产”。

这个项目就是为了解决这件事。它要把一条研究链路真正打通：

1. 从真实 3GPP 归档中提取正文和过程切片。
2. 把过程切片转成结构化状态机。
3. 验证状态机和消息约束是否可信。
4. 基于状态机规则生成序列级测试。
5. 判定是否出现协议逻辑漏洞。
6. 输出可以回放、可以复现、可以继续接入目标系统的测试资产。

### 1.2 这个项目不做什么

- 不把大模型回答直接当作漏洞结论。
- 不把“能编码一条消息”误当成“已经完成协议安全测试”。
- 不把 README 写成脚本清单或一次性实验笔记。

### 1.3 新人如何阅读本手册

如果你是第一次接手这个工程，推荐按这个顺序看：

1. 第 3 章：先看一条端到端真实样例，建立整体感觉。
2. 第 2 章：再看整体架构，理解每一层负责什么。
3. 第 4 章：看当前做到了什么、还缺什么。
4. 第 5 章：本地重跑一次主链路。
5. 第 6 章：最后再进入文件目录和脚本实现。

### 1.4 当前研究范围

当前仓库聚焦的规范与过程如下：

- 规范版本：`TS 38.331 Rel-17`
- 当前已覆盖过程：
  - `5.3.3` RRC connection establishment
  - `5.3.5.11` full configuration
  - `5.3.13` RRC connection resume

当前保留的两条主线：

- 真实规范接入与可信建模
- 协议状态机逻辑漏洞测试

## 2. 技术选型与整体架构

### 2.1 设计原则

这个项目遵循四条原则：

1. 真实资料优先  
   不依赖示例化玩具语料，直接以真实规范归档为起点。

2. 大模型只负责语义提取  
   文档导入、约束验证、逻辑判定、结果落盘全部由本地代码完成。

3. 任何漏洞候选都必须可复核  
   需要能回到规则、轨迹和产物上，而不是停留在自然语言解释。

4. 任何关键输出都必须能进入下一层  
   状态机不是最终成果，它必须能继续进入消息编码、逻辑测试和执行复现。

### 2.4 分层职责表

相比只看图，新人通常更需要知道“每一层到底负责什么、输入什么、输出什么”。下面这张表是整个项目最重要的阅读入口。

| 层次 | 核心职责 | 主要输入 | 主要输出 | 不负责什么 |
| --- | --- | --- | --- | --- |
| 规范输入与资料治理层 | 把原始归档整理成可消费资料 | 3GPP 归档、规则文件 | 标准化主文档、过程切片、导入报告 | 不负责漏洞判断 |
| 协议语义提取层 | 把章节文本转成结构化协议模型 | 过程切片、结构约束 | 状态机、消息清单、定时器清单、变量清单 | 不负责结果定性 |
| 模型校验与消息约束层 | 把模型从“看起来合理”推进到“可验证可编码” | 状态机、章节文本、ASN.1 定义 | 校验报告、ASN.1 定义、编码验证结果 | 不负责生成漏洞结论 |
| 逻辑漏洞判定层 | 依据规则和轨迹识别逻辑异常 | 已校验状态机、逻辑规则、扰动策略 | 错误接受、状态偏差、规则违反、漏洞候选 | 不负责重新解释规范 |
| 序列测试与执行复现层 | 把测试输入落成可执行样本和可复现证据 | 状态机、消息约束、逻辑规则 | 种子样本、协议二进制、抓包样本、回放脚本、覆盖配置 | 不负责重新建模 |

### 3.2 这个样例要回答什么问题

我们选择 `5.3.13 RRC connection resume`，因为它同时覆盖：

- 真实过程切片
- 状态机提取
- 状态机校验
- 逻辑规则判定
- 跨过程消息混淆
- 运行时复现资产生成

这个样例重点观察的问题是：

**如果在连接恢复过程中错误地收到 `RRCSetup`，目标会不会把它当成合法消息接受，从而错误进入 `RRC_CONNECTED`。**

| 阶段 | 文件 | 作用 |
| --- | --- | --- |
| 真实切片 | `corpus/real/slices/5.3.13-rrc-connection-resume.md` | 连接恢复过程原始切片 |
| 状态机模型 | `outputs/efsm_real/5.3.13-rrc-connection-resume.json` | 提取出的结构化状态机 |
| 校验报告 | `outputs/reports_real/5.3.13-rrc-connection-resume.json` | 验证模型结构、术语和引用 |
| 逻辑规则 | `constraints/logic_rules.yaml` | 定义不变量和扰动策略 |
| 测试轨迹 | `outputs/logic/traces/5.3.13-resume_nominal-cross_procedure_response_swap.json` | 记录这条异常样例的完整执行过程 |
| 测试摘要 | `outputs/logic/summary.json` | 汇总整个逻辑测试批次 |
| 运行时种子 | `outputs/runtime/seeds/5.3.13-rrc-connection-resume.json` | 连接恢复过程对应的运行时样本 |
| 运行时产物 | `outputs/runtime/manifest.json` | 汇总所有二进制、抓包和回放入口 |

### 3.4 这条样例是怎么生成的

#### 第一步：从真实规范中切出过程

真实归档先被解析成主文档，再切出连接恢复这一段过程文本。  
这一层的结果是：

- 有真实正文
- 有过程级切片
- 有导入统计

#### 第二步：把过程切片提取成状态机

大模型读取连接恢复切片后，输出结构化状态机。  
这里产出的不是摘要，而是后续所有测试都要消费的模型文件：

- 状态
- 事件
- 条件
- 动作
- 引用条款

#### 第三步：校验这个状态机是否可信

模型提取出来后不会直接进入测试，而是先经过两类检查：

- 状态机结构和术语覆盖检查
- ASN.1 提取、编译和编码往返检查

只有通过这层，后面生成出来的样本才有工程意义。

#### 第四步：构造一条跨过程混淆测试

逻辑测试批次中有一类扰动叫“跨过程消息混淆”。  
在这条样例里，系统故意在连接恢复过程中投放了属于连接建立过程的 `RRCSetup`。

这条样例对应的轨迹文件是：

- `outputs/logic/traces/5.3.13-resume_nominal-cross_procedure_response_swap.json`

这条轨迹当前记录的关键结果是：

- 参考执行模型最终保持在 `RRC_INACTIVE`
- 候选目标错误进入 `RRC_CONNECTED`
- 命中了 `4` 条异常：
  - 错误接受
  - 状态偏差
  - 缺少 `RRCResume` 仍进入终态
  - 错误接受了不该在该过程里出现的 `RRCSetup`

### 3.5 这条样例最终产出了什么

这条样例最后不会只留下一条“怀疑有问题”的文本说明，而是会形成一整组证据：

1. 过程切片  
   说明测试基于哪段真实规范。
2. 状态机模型  
   说明系统认为协议应当如何执行。
3. 校验报告  
   说明这个模型不是随意生成的。
4. 逻辑测试轨迹  
   说明异常发生在哪一步。
5. 协议二进制和抓包样本  
   说明后续可以继续做回放和目标系统接入。

- 当前逻辑测试仍以离线执行和离线对比为主。
- 当前还没有接入真实协议处理器或在线 OAI 注入。

## 8. 附录：术语

### 8.1 语义抽取与模型

- 扩展有限状态机（EFSM）
  - 相比普通状态机，多了条件、动作、定时器和上下文变量。

- 守卫条件（Guard）
  - 一条状态转移成立前必须满足的条件。

- 动作（Action）
  - 状态转移时执行的副作用，例如启动定时器、发送消息、更新变量。

- 上下文变量（Context Variable）
  - 会影响协议执行路径的变量，例如某个过程是否已触发、某个标识是否存在。

### 8.2 逻辑漏洞挖掘

- 测试批次（Campaign）
  - 一次成批执行的测试活动，包含一组用例、统一规则和汇总结果。

- 参考执行模型（Reference Model）
  - 项目内部从真实状态机推导出的基准执行模型，用来表示“协议应该如何运行”。

- 候选目标（Candidate Target）
  - 与参考执行模型进行对比的目标对象，可能是真实实现，也可能是当前这版中的本地宽松模型。

- 判定器（Oracle）
  - 用于判断一次测试是否出现异常的规则集合。

- 不变量（Invariant）
  - 在合法执行中必须始终成立的约束。

- 正常路径（Nominal）
  - 符合规范主流程的标准执行序列。

- 边界样本（Boundary）
  - 用于覆盖字段边界条件的样本。

- 扰动（Perturbation）
  - 对正常序列施加的变化，例如重复、乱序、插入超时、跨过程替换。

- 错误接受（Negative Acceptance）
  - 目标错误地接受了一条本应拒绝的消息或序列。

- 异常拒绝（Unexpected Rejection）
  - 目标错误地拒绝了一条本应接受的消息或序列。

- 状态偏差（State Divergence）
  - 参考执行模型与候选目标在同一步输入后进入了不同状态。

- 执行轨迹（Trace）
  - 某个测试用例从开始到结束的逐步执行记录。

- 结果归因（Triage）
  - 对异常结果做归类、去重、复现整理的过程。

### 8.3 编码与复现

- 抽象语法记法一（ASN.1）
  - 规范里用于描述消息结构的标准表示法。

- 非对齐压缩编码规则（UPER）
  - ASN.1 的一种编码方式，本项目当前用它生成 RRC 相关二进制。

- 协议数据单元（PDU）
  - 编码后的协议二进制消息。

- 抓包样本（pcap）
  - 用于保存回放数据的抓包文件。

- 回放（Replay）
  - 把已经生成的样本再次发给目标系统，用于复现。

- 编码往返校验（Roundtrip）
  - 先编码，再解码回来，检查结果是否一致。

### 8.4 目标系统与配套工具

- 大语言模型（LLM）
  - 本项目里主要用于提取协议语义，不直接负责漏洞定性。

- OAI
  - `OpenAirInterface`，开源移动通信协议栈/实验平台。

- 适配层（Adapter）
  - 把项目内部的数据结构接到具体目标系统上的转换层。

- 测试连接层（Harness）
  - 负责把生成的样本真正送入待测对象的连接层。
